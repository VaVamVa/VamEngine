## Q1: Framework.lib 링크 오류

**질문:**
`Program\stdafx.h`에서 `Framework.lib`를 링크하려 하는데 안 됨.

**답변:**
두 가지 경로 불일치 문제.

1. **헤더 경로 문제**
   - `AdditionalIncludeDirectories` → `$(SolutionDir)_Libraries` (`_Libraries/Framework.h` 탐색)
   - 실제 `Framework.h` 위치 → `$(SolutionDir)Engine\`

2. **라이브러리 경로 문제**
   - `AdditionalLibraryDirectories` → `$(SolutionDir)_Libraries\` (`_Libraries/Framework.lib` 탐색)
   - 실제 `.lib` 위치 → `$(SolutionDir)_Libraries\Framework\x64\Framework.lib`

**수정 (Debug|x64 기준):**
- `AdditionalIncludeDirectories`에 `$(SolutionDir)Engine` 추가
- `AdditionalLibraryDirectories`를 `$(SolutionDir)_Libraries\Framework\x64\`로 변경

## Q2: 모바일 대응 시 Program 템플릿 문제

**질문:**
모바일 환경도 목표라면, Program 프로젝트를 Windows Desktop App 템플릿으로 만들면 안 되었나?

**답변:**
템플릿 선택 자체는 큰 문제가 아님. 핵심은 아키텍처 설계에 있음.

크로스 플랫폼 엔진의 핵심 원칙: **플랫폼 레이어(Program)와 엔진 코어(Engine)의 완전한 분리**.

```
[Windows Program]  [Android Activity]  [iOS UIApplication]
        ↓                  ↓                    ↓
        └──────────[Engine Core Library]────────┘
                   (플랫폼 코드 없음)
```

`Program` 프로젝트는 어차피 Windows 전용 부트스트랩 코드이므로, 모바일로 가면 이 프로젝트 자체를 교체함. Windows Desktop 템플릿이어도 무방.

**실제로 중요한 것:** Engine 라이브러리 안에 플랫폼 종속 코드가 없어야 함.
DirectX12 자체가 Windows 전용이므로, 렌더링 백엔드를 인터페이스로 추상화하는 것이 진짜 과제.

```cpp
class IRenderDevice { ... };              // 플랫폼 무관
class DX12RenderDevice : IRenderDevice    // Windows
class VulkanRenderDevice : IRenderDevice  // Android/Linux
class MetalRenderDevice : IRenderDevice   // iOS
```

## Q3: 플랫폼별 프로젝트 분리 여부

**질문:**
하나의 Engine 프로젝트가 각 플랫폼으로 대응된다면, 플랫폼별 프로젝트가 따로 존재해야 하나?

**답변:**
반드시 분리할 필요는 없음. 세 가지 방식 존재.

1. **단일 프로젝트 + 조건부 컴파일** — `#ifdef _WIN32` 등으로 파일 분기. 단순한 엔진에 적합.
2. **플랫폼별 프로젝트 분리** — `Engine.Core` / `Engine.Windows` / `Engine.Android` 등으로 명시적 분리.
3. **CMake (권장)** — 프로젝트는 하나, CMake가 플랫폼마다 다른 빌드 파일 생성.
   ```cmake
   if(WIN32)
       target_sources(Engine PRIVATE ${WIN32_SOURCES})
   elseif(ANDROID)
       target_sources(Engine PRIVATE ${ANDROID_SOURCES})
   endif()
   ```
   Unreal Engine 등 대부분의 상용 엔진이 이 방식.

**권장 디렉토리 구조:**
```
Engine/
├── Core/           ← 플랫폼 무관 (인터페이스, 수학, 메모리)
└── Platform/
    ├── Windows/    ← DX12, Win32 윈도우
    └── Vulkan/     ← 추후 Android/Linux 대응
```

## Q4: 크로스 플랫폼 조건에서 Program 프로젝트 구성

**질문:**
- Program 프로젝트 하나에 게임 제작
- 플랫폼별 Framework에서 플랫폼 대응
- 어떤 플랫폼이든 Program Source로 같은 화면/상태 유지

위 조건을 만족하려면 Program 프로젝트는 무엇으로 만들어야 하나?

**답변:**
**Program은 Static Library**여야 함.

진입점(Entry Point)은 Framework가 가지며, Program은 호출됨.

```
[Windows]  WinMain()              ┐
[Android]  ANativeActivity_onCreate()  ┤ → Framework → Program.lib
[iOS]      UIApplicationMain()         ┘
```

**등록 패턴:**
```cpp
// Framework가 정의 (Program이 구현)
class IApplication {
public:
    virtual void OnInit()    = 0;
    virtual void OnUpdate()  = 0;
    virtual void OnRender()  = 0;
    virtual void OnDestroy() = 0;
};

// Program이 구현
class MyGame : public IApplication { ... };

// Framework에 등록
IApplication* CreateApplication() {
    return new MyGame();
}
```

**vcxproj 변경:**
```xml
<!-- Application → StaticLibrary -->
<ConfigurationType>StaticLibrary</ConfigurationType>
```

| | Program | Framework |
|---|---|---|
| **타입** | Static Library | Executable / Shared Lib |
| **역할** | 게임 로직, 씬, 오브젝트 | 창 생성, 입력, 렌더링 백엔드 |
| **플랫폼 API** | 없음 | Win32 / Android NDK / UIKit |
| **진입점** | `CreateApplication()` 반환 | `WinMain` / `main` |

## Q5: 상용 엔진의 크로스 플랫폼 방식

**질문:**
상용 엔진에서는 어떤 방식으로 크로스 플랫폼을 처리하나?

**답변:**

**Unreal Engine — 동적 모듈 시스템**
- 플랫폼별 Launch 모듈이 진입점 (`LaunchWindows.cpp` → `WinMain`)
- 게임은 `.dll`/`.so`로 빌드되어 런타임에 동적 로드
- `RHI(Render Hardware Interface)`로 DX12/Vulkan/Metal 추상화
- `UBT(Unreal Build Tool)`이 플랫폼별 빌드 자동화

**Godot — 플랫폼 OS 클래스 + 공통 Main**
```cpp
// platform/windows/godot_windows.cpp
int WINAPI WinMain(...) {
    OS_Windows os;    // 플랫폼 구현체
    Main::setup();    // 공통
    Main::start();    // 공통
    os.run();         // 플랫폼별 메인루프
    Main::cleanup();  // 공통
}
```

**Unity — 스크립팅 VM**
- C++ 런타임 위에 C# VM(Mono/IL2CPP)을 올리는 방식
- 게임 스크립트(C#)는 완전히 플랫폼 무관

**현재 엔진에 적합한 참고:** Godot 방식 (Program.lib = Godot의 Main, Framework = Godot의 OS 클래스)

## Q6: 게임 실행파일 vs 엔진 실행파일

**질문:**
Program이 Static Library이고 Framework가 실행파일이라면, 내 게임이 아닌 엔진이 실행파일이 되는 것 아닌가?

**답변:**
맞음. 그리고 그게 맞는 방향.

```
개발 중:  Framework.exe  (엔진 실행파일)
              ↑ 링크
          Program.lib    (게임 코드)

배포 시 (Static 링크):
          MyGame.exe  ← 이름만 바꾼 것
          = Framework 런타임 + Program 게임코드가 하나로 합쳐진 것
```

Unreal도 동일: 패키징 후 `MyGame.exe`는 UE4 런타임 + MyGame 모듈의 합본.

**결론:**
> 게임 실행파일 = 엔진 런타임이 특정 게임을 실행하도록 구성된 것

엔진과 게임은 코드 레벨에서 분리되어 있지만, 배포 레벨에서는 하나의 실행파일로 합쳐짐.
이 구조 덕분에 같은 `Program.lib`를 Windows/Android/iOS Framework에 각각 링크하면 플랫폼별 게임 빌드가 만들어짐.

## Q7: Program 정적 vs 동적 라이브러리

**질문:**
엔진 실행파일 내부에서 Program 라이브러리를 링크할 때, 정적 라이브러리와 동적 라이브러리 중 무엇이 좋은가?

**답변:**
**정적 라이브러리 권장.**

| | 정적 (.lib/.a) | 동적 (.dll/.so) |
|---|---|---|
| **배포** | 단일 실행파일로 합쳐짐 | .dll 별도 배포 필요 |
| **크로스 플랫폼** | 플랫폼별 Framework에 그냥 링크 | 플랫폼마다 .dll/.so/.dylib 관리 |
| **빌드** | 변경 시 전체 재링크 | 게임 코드만 재빌드 가능 |
| **Hot Reload** | 불가 | 가능 (런타임 교체) |
| **ABI 의존성** | 없음 | 컴파일러/버전 맞춰야 함 |

**정적이 유리한 이유 (현재 구조 기준):**
```
[Windows Framework.exe]  ← Program.lib 링크
[Android Framework]      ← Program.a  링크
[iOS Framework]          ← Program.a  링크
```
동적으로 가면 플랫폼마다 로딩 방식이 달라 복잡도가 올라감.

**동적이 유리해지는 시점:** 개발 규모가 커져 Hot Reload(게임 코드만 변경 즉시 반영)가 필요해질 때.
Unreal의 Live Coding이 이 방식. 초기 설계부터 넣을 이유는 없으며 필요 시 전환 고려.

**결론:** 정적 라이브러리로 시작, Hot Reload가 필요해지면 그때 동적으로 전환 고려.

## Q8: 단일 프로젝트에서 CMake 빌드 자동화 가능 여부

**질문:**
단일 프로젝트 내에서도 CMake로 빌드 자동화가 가능한지?

**답변:**
가능. CMake는 프로젝트 분리와 무관하게 사용할 수 있음.

```cmake
add_library(VaEngine STATIC
    Core/Math.cpp
    RHI/DX12/DX12Device.cpp
    Platform/Windows/WindowsEngine.cpp
)

if(WIN32)
    target_sources(VaEngine PRIVATE Platform/Windows/WindowsEngine.cpp)
endif()

target_compile_features(VaEngine PUBLIC cxx_std_20)
target_include_directories(VaEngine PUBLIC ${CMAKE_SOURCE_DIR}/Engine)
```

**실질적 이점:**
- VS 프로젝트 파일 직접 편집 불필요
- 플랫폼별 빌드 설정 한 곳에서 관리
- 나중에 모듈 분리 시 `add_library` 하나 추가로 점진적 확장 가능

| | VS 프로젝트 직접 관리 | CMake |
|---|---|---|
| 초기 설정 | 간단 | CMakeLists.txt 작성 필요 |
| 파일 추가 | .vcxproj 직접 편집 | CMakeLists.txt만 수정 |
| 플랫폼 확장 | 설정 복잡해짐 | 조건문으로 처리 |
| 나중에 모듈 분리 | 대규모 재설정 | 점진적으로 가능 |

**결론:** 크로스 플랫폼이 목표라면 지금 CMake를 도입해두는 것이 장기적으로 유리.

## Q9: 디렉토리 분리 후 CMake 추후 도입 시 작업량

**질문:**
디렉토리만 엄격하게 분리해두면 CMake를 추후에 도입하더라도 작업량이 많지 않은가?

**답변:**
맞음. CMake 도입 시 작업의 대부분은 CMakeLists.txt 작성이지 소스 코드나 디렉토리 이동이 아님.

디렉토리가 명확하면 glob으로 한 번에 수집 가능:
```cmake
file(GLOB_RECURSE CORE_SOURCES     "Core/*.cpp")
file(GLOB_RECURSE RHI_SOURCES      "RHI/*.cpp")
file(GLOB_RECURSE PLATFORM_SOURCES "Platform/Windows/*.cpp")

add_library(VaEngine STATIC ${CORE_SOURCES} ${RHI_SOURCES} ${PLATFORM_SOURCES})
```

반면 디렉토리가 뒤섞여 있으면 파일을 하나씩 열거하거나 이동해야 해서 작업량이 커짐.

**결론:** 지금 디렉토리 구조만 잘 잡아두면 CMake 도입은 CMakeLists.txt 파일 몇 개 작성으로 끝남. VS 프로젝트로 시작해도 무방.

## Q10: 최종 프로젝트 구조 확정

**질문:**
Engine, Program Logic, Platform Entries 구조로 프로젝트를 나누는 것이 맞는가?
Android 템플릿도 지금 잡아두는 게 좋은가?

**답변:**
맞음. 최종 구조:

```
Engine.lib      ← 렌더러, RHI, 기본 오브젝트, IApplication 인터페이스 정의
Game.lib        ← 게임 로직, IApplication 구현 (플랫폼 코드 없음)

Platform/
├── Windows.exe ← WinMain → Engine 초기화 → Game 실행
└── Android     ← ANativeActivity → Engine 초기화 → Game 실행
```

**빌드 의존성:**
```
Engine.lib ← Game.lib ← Windows.exe
                      ← Android
```

**Android를 지금 잡아두는 이유:**
양쪽 플랫폼 진입점을 보면서 인터페이스를 설계해야 진짜 플랫폼 무관한 IApplication이 나옴.
Android가 없으면 Windows에 종속된 인터페이스가 될 위험이 있음.

**각 프로젝트 타입:**
| 프로젝트 | VS 템플릿 | 역할 |
|---|---|---|
| Engine | Static Library | 렌더러, 기본 오브젝트, 인터페이스 |
| Game | Static Library | 게임 로직, 게임 오브젝트 |
| Windows | Windows Desktop App | WinMain (얇은 진입점) |
| Android | Android 템플릿 | ANativeActivity (얇은 진입점) |
